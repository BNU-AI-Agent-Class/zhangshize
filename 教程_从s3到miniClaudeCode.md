# 从 s3 到 mini Claude Code — 逐阶段教程

> 读法:每个阶段先**跑**(照"运行验收"做),再**读**代码,最后想**思考题**。
> 每个文件都是完整可跑的;每一步相对上一步的改动都很小——差异就是这一阶段的"新想法"。

---

## 先认识练习靶子:demo_project(为什么要用它?)

每个阶段都有两个例子:一个**简单**、一个**复杂**。简单例子里,你多半会觉得"这机制没啥用";
只有在**复杂、真实**的项目上,你才会看到它为什么不可或缺——这正是本教程的核心:**痛点驱动**。

问题是,"创建 hello.txt"这种一次性任务撑不起"复杂"。所以我们准备了 `demo_project/`:
一个**麻雀虽小、五脏俱全**的玩具笔记本 CLI(`notes_app`,多文件互相调用 + README + TODO + 一个故意藏起来的 bug),
像个真实小项目。**每个阶段的"🔥 复杂例子"和所有作业,都在这同一个靶子上跑**——人人一致、可复现,
让你亲眼看到:同一个机制,简单任务用不上,一到真实项目就离不开。

开动前先逛一圈(2 分钟):

```bash
cd demo_project
cat README.md              # 看它是个什么项目
python -m notes_app.cli list
python -m notes_app.cli stats
```

## 起点回顾:昨天的 s3

`s3_agent.py` 已经是一个真 agent 了:双层循环 + 文本协议("命令:XXX"/"完成:XXX")+ `os.popen` 执行。
它的两个"软肋",正是今天要补的地方:

- **协议太脆**:靠 `split("命令:")` 切字符串,AI 一不守格式就 `IndexError`(昨天的彩蛋)。
- **工具太少**:只有一个 `os.popen`,没有分工、没有计划、没有安全约束。

真 Claude Code 强在哪?就强在它把这两点做扎实了。我们一步步来。

## c0 — JSON 协议:让机器听得懂机器

**目标**:把文本协议换成结构化的 JSON,解析从"切字符串"变成"读字段"。

**运行验收**
```bash
python c0_json.py
# 你：在当前目录创建 hello.txt 内容为 hi，然后列出 txt 文件
# 预期:看到 [执行] echo ... → [结果] hello.txt → [完成] ...
```

**新想法**:AI 现在回复 `{"tool": "bash", "args": {"cmd": "ls"}}` 或 `{"done": "..."}`。
第 12 行 `json.loads(reply)` 把这段文本变成 Python 字典,第 14 行直接 `action["args"]["cmd"]` 取命令——
再也不用数"命令:"在第几个字符了。这就是为什么所有真 agent 都用结构化协议。

**和 s3 比**:双层循环一字没变,变的只有"协议"和"解析那一行"。

**🔗 在真 Claude Code 里,这就是「tool_use 结构化输入」**
真 Claude Code 里,模型不是吐一段文字让你切,而是直接返回结构化的工具调用(tool_use)。我们手写的 JSON,就是它的极简版。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `列出 demo_project 里所有文件` | 一句命令、没引号没换行,昨天 s3 的文本协议也能切对——这时你看不出 JSON 有啥好。 |
| 🔥 复杂·痛点凸显 | `在 demo_project 写一个 hello.py，内容是 print("你好, world")` | 内容里一带引号、冒号、换行,s3 的 split("命令:") 就切歪了。JSON 用 {} 把"做什么"和"内容"分得清清楚楚——这就是结构化协议的意义。 |

**思考题**
> 让 c0 往文件里写一段带很多引号的 Python 代码,看它出没出错。为什么结构化 JSON 能避免昨天 s3 的切割歧义?

## c1 — 工具箱:不止一把锤子

**目标**:给 agent 多个专用工具(read_file / write_file / bash),按名字派发。

**运行验收**
```bash
python c1_toolbox.py
# 你：把"你好世界"写进 greet.txt，再读回来给我看
# 预期:[调用] write_file(...) → [调用] read_file(...) → [完成]
```

**新想法**
- 三个工具函数 + 一个 `TOOLS` 字典(名字 → 函数)。
- `TOOLS[name](**args)`:AI 说用哪个工具,代码就调哪个——这叫"派发(dispatch)"。
- **顺手修好了 c0 的崩溃**(两道保险):① `parse()` 先剥掉 AI 可能加的 ` ``` ` 包裹再解析;② 万一还是坏 JSON(比如字符串里混进了英文双引号),`try/except` 会接住它、请模型重发,而不是让程序死掉。这就是你在 c0 思考题里想到的那些坑——从这里开始,协议"能用"了。

**为什么真 Claude Code 有专门的 Read/Write,而不全用 bash?**
> 专用工具更**可控、可审计**:它能限制路径、记录每一次读写、避免 `cat`/`echo` 里的转义地狱。
> 这是"能力"和"安全"的平衡——你后面写自己的 agent 会反复遇到。

**🔗 在真 Claude Code 里,这就是「Read / Write / Bash 工具」**
真 Claude Code 有几十个专用工具。为什么不全用 bash?因为专用工具能限制路径、记录每次读写、避开 shell 转义地狱——更可控、可审计。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `把「测试」写进 demo_project/tmp.txt` | 就写一个文件,bash 的 echo 也能干,read_file / write_file 好像多此一举。 |
| 🔥 复杂·痛点凸显 | `把 demo_project/TODO.md 第一条待办从 [ ] 改成 [x]` | 要精确改一行,用 bash 的 sed 得跟转义、特殊字符搏斗,还可能误伤别的行;read_file 读出来→想清楚→write_file 写回去,干净可控。这就是 CC 有专门 Read/Write 的原因。 |

**思考题**
> 同样是改 TODO.md 一行,让 c1 用 write_file 做一次。再想:如果改用 bash 的 sed,遇到行里有特殊符号会出什么岔子?

## c2 — 计划是一等公民:TodoWrite

**目标**:让 agent 先列计划、边做边勾,把"思考过程"变得可见。

**运行验收**
```bash
python c2_todo.py
# 你：创建 a.txt 内容为1，再创建 b.txt 内容为2，然后列出所有txt
# 预期:先打印一张 ☐ 清单,每完成一步 ☐ 变 ☑
```

**新想法**:`todo_write` **也只是一个普通工具**(第 7 行)——它不执行命令,只是更新并打印一张清单。
系统提示(第 9 行)要求 AI"接到多步任务先列计划"。你会看到 agent 变得有条理了。

**关键洞察**:Claude Code 的 TodoWrite 看起来很高级,本质就是"让模型把计划写进一个它能反复更新的工具"。
计划可见,人就能盯着它别跑偏——这也是"人在环"的一部分。

**🔗 在真 Claude Code 里,这就是「TodoWrite 工具」**
你用 Claude Code 做多步任务时,它会先列一张待办清单、逐条打勾。那就是 TodoWrite——和我们这几行本质一样:让模型把计划写进一个能反复更新的工具。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `给 demo_project 加一个空文件 draft.txt` | 一步就完事,列不列计划都一样。 |
| 🔥 复杂·痛点凸显 | `给 notes_app 加一个 export 命令，把所有笔记导出成 markdown` | 这要读 cli.py、改 cli.py、可能加函数、再测试——好几步。没有计划,agent 容易漏掉"测试"或改错顺序。todo_write 让它先排步骤、边做边勾,你也能盯着它别跑偏。 |

**思考题**
> 让 c2 做"给 notes_app 加 export 命令",看它列了哪几步。如果不让它先列计划,你猜它最容易漏掉哪一步?

## c3 — 子智能体:把脏活外包出去

**目标**:主 agent 能派一个"子 agent"独立完成一件事,只拿回结论。

**运行验收**
```bash
python c3_subagent.py
# 你：派一个子助手统计当前目录有几个 .py 文件，只告诉我数字
# 预期:[调用] subagent(...) → [结果] 6 → [完成]
```

**新想法**:第 7–9 行的 `subagent` 函数,内部**新开一份全新的 `messages`** 跑它自己的循环,
最后只 `return` 一句结论。主 agent 的对话历史因此保持干净——不会被"翻了 200 个文件"的过程塞爆。

**这解决了什么?**
> 长任务最大的敌人是"上下文越来越脏"。把探索性的脏活丢给子 agent,主线只留结论,
> 就像你写论文时让室友去查一堆资料,他只把结论发你,而不是把 100 篇论文全塞你脑子里。

**🔗 在真 Claude Code 里,这就是「Task / 子 agent」**
在 Claude Code 里让它"探索这个代码库",它会派出子 agent(Task 工具)去翻文件,只把结论带回来。你的主对话因此不会被一大堆文件内容淹没。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `数一下 demo_project 有几个 .py 文件` | 一条命令就出数字,用不用子 agent 没差别。 |
| 🔥 复杂·痛点凸显 | `读懂整个 demo_project 的架构，一句话总结每个 .py 文件` | 这要翻开好几个文件。全塞进主对话,上下文很快被几十屏代码撑爆,后面就变慢变贵变糊涂。派子 agent 去读、只拿回一段总结,主线保持干净——这正是 CC 探索大项目的办法。 |

**思考题**
> 让 c3 去"读懂 demo_project 架构"。看子 agent 返回的是全部文件内容,还是一段总结?为什么后者对长任务这么关键?

## c4 — 上下文压缩:记不下就做笔记

**目标**:对话太长时,自动把历史折叠成一条摘要,重开窗口。

**运行验收**
```bash
python c4_compact.py
# 连续下几个命令(比如依次创建 5 个文件),当历史超过阈值,会看到 [压缩]
# 想快点看到效果:临时把第 5 行 LIMIT 改小(比如 4)
```

**新想法**:第 7 行 `compact()` 让模型**给自己的对话历史写个摘要**,
然后用 `[系统提示 + 摘要]` 当新的起点(第 8 行)。旧的一大堆细节被"折叠"掉了,但关键进展留下了。

**和昨天的联系**:这正是 Day1 讲的"上下文腐化(context rot)"的解药之一。
真 Claude Code 按 token 数触发,我们这里简化成"消息条数",原理一样。

**🔗 在真 Claude Code 里,这就是「/compact · 自动压缩」**
Claude Code 里你能敲 /compact,长会话它也会自动压缩历史。原理就是让模型给之前的对话写个摘要、重开窗口——和我们这段 compact() 一样。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `连着问它两三个小问题` | 对话短,历史没多少,压不压缩看不出区别。 |
| 🔥 复杂·痛点凸显 | `在 demo_project 里一步步排查 stats 的 bug（来回很多轮）` | 真实调试常常十几二十轮,历史越堆越长。不压缩,迟早撑爆上下文窗口、越来越慢越来越贵。压缩把旧对话折叠成一条摘要,留住进展、丢掉啰嗦。(想快点看到:把 LIMIT 改小) |

**思考题**
> 把 LIMIT 改小,在 demo_project 上多聊几轮触发 [压缩]。压缩后它还记得任务目标吗?如果摘要漏了个关键细节会怎样?

## c5 — 权限门:危险动作先问人

**目标**:执行危险命令(rm/sudo/mv…)前,停下来征求人的同意。

**运行验收**
```bash
python c5_guard.py
# 你：用 rm 删除某个文件
# 预期:弹出 [⚠ 权限] ... 允许吗？(y/n)  —— 输入 n,命令不执行
```

**新想法**:第 4 行一个危险词黑名单,第 5–8 行的 `bash` 在执行前先检查——
命中危险词就 `input()` 问人,人说 `n` 就不执行,并把"用户拒绝了"告诉 AI。这就是 **Human-in-the-loop**。

**这是终点**:c5 只有 ~35 行,但它已经有了真 Claude Code 的五根支柱雏形——
**工具箱、计划、子 agent、压缩、权限**。恭喜,你造出了一个 mini Claude Code。

**🔗 在真 Claude Code 里,这就是「危险操作确认弹窗」**
Claude Code 执行 rm、改关键文件、跑未知脚本前,会弹出来问你要不要允许。那就是权限系统——我们用一个黑名单 + input() 做了它的极简版。

**痛点对比(都在 demo_project 上试)**

| | 命令 | 说明 |
|---|---|---|
| 😌 简单·看不出必要 | `列出 demo_project/logs 里的文件` | 只是看看,没有风险,问不问都行。 |
| 🔥 复杂·痛点凸显 | `清理 demo_project 里的临时文件` | "清理"听着无害,但 agent 可能生成 rm 命令,一不小心就删到你不想删的文件。没有权限门,它手一滑就把你的东西删了;有了门,危险命令先停下来问你 y/n。 |

**思考题**
> 让 c5"清理 demo_project 的临时文件",在它要 rm 时输入 n。想一想:黑名单靠关键词匹配,rm 能不能被写成绕过检查的样子?
